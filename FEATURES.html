🔐 Overview & who uses it

What the application is for.

What it is

A Slack app that turns Slack's raw channels and memberships into a governed access-review programme: it scores risk, runs periodic reviews with sign-off, enforces removals, records a tamper-evident evidence trail, and lets members request access through an approval flow — the governance layer Slack itself doesn't provide.

How it works

  • Admins open the app's Home tab to get the dashboard (members, channels, risk, and all tools).
  • Members open the same app and get a lightweight home where they can request channel access.
  • The app is multi-workspace: each installing workspace has its own bot token and its data is isolated by team_id.

📸 Access snapshot & risk scoring

The data foundation every other feature reads from.

What it is

A point-in-time map of the workspace: every user, every channel, who's in each channel, and a 0–100 risk score per channel.

How it works

  • Pulls users and channels, then each channel's members, and builds a per-user access map.
  • Channel risk is a weighted blend of external users, guests, privileged users (owners/admins), deactivated members, and whether the channel name looks sensitive (finance, hr, legal, exec, security…). Bands: 🔴 Critical ≥80, 🟠 High ≥60, 🟡 Medium ≥40, 🟢 Low.
  • Cached briefly so the dashboard, modals, and exports reuse one scan; invalidated after a revoke or a domain change.
Private-channel reality: the app only sees private channels the bot is a member of. Public channels are always visible.

⚙️ Internal / external domains

Admin Dashboard → ⚙️ Domains

What it is

Decides which email domains count as internal; everyone else is external. This drives risk scores and the "external" flags in every report.

How it works

  • The modal lists every email domain found in the workspace with member counts (most common first). Tick = internal, untick = external.
  • Pre-ticked with your current setting, or the auto-detected majority domain if you haven't set one. A free-text box adds domains not in the list.
  • Precedence: in-app setting → INTERNAL_EMAIL_DOMAINS env → majority auto-detect. Tick nothing to return to auto-detect.
  • Saving clears the snapshot cache; press Refresh to recompute flags. Stored per workspace.
Auto-detect picks only the single most common domain — so a workspace full of personal Gmail addresses would wrongly treat gmail as "internal". Set your real domain here to fix it.

📋 Access review campaigns

Admin launches Auto-completes

What it is

A structured certification: for a chosen scope of channels, each channel's reviewer decides Keep / Remove / Flag for every member, and every decision is recorded as evidence.

How it works

  • Launch with a name, scope (all / private only / high-risk ≥70), due date, and recurrence (one-off / monthly / quarterly).
  • Each channel's reviewer is its creator (or you, if the creator has left). Reviewers get a DM with a deep-link to their queue.
  • Decisions are saved to the campaign and chained into the audit log. The campaign auto-completes when every membership is decided, and the creator is notified.
  • Recurring campaigns automatically spawn the next cycle when they complete or lapse.
A campaign "Remove"/"Flag" is a decision/recommendation, not an enforcement — it does not kick anyone. Enforce via Revocation from the dashboard.

✅ Reviewer queue & bulk decisions

Built for hundreds of members per channel.

What it is

A two-level App Home surface that makes reviewing large channels practical instead of clicking a menu on every person.

How it works

  • Level 1 — channel index: your channels with progress bars and "needs attention" counts.
  • Level 2 — roster: a filterable, paginated member list. Filters (Undecided / Needs attention / All), a page-size selector (10 / 25 / 50), and checkboxes.
  • Bulk actions: Select all on page then Keep / Revoke / Flag selected (Remove/Flag ask for one shared justification). Keep all remaining routine clears every non-flagged member in one click, leaving only the exceptions.

⏰ Reminders & escalation

Runs daily

What it is

Automatic nudges so reviews actually get finished on time.

How it works

  • A daily sweep finds active campaigns near or past their due date with pending channels.
  • Reviewers with unfinished channels get a reminder DM (channels + a one-click queue link).
  • When a campaign is overdue, the creator gets an escalation DM naming how many reviewers are still pending.
  • Runs on an interval only, so a redeploy never re-sends. Lead time tunable via REVIEW_REMINDER_LEAD_DAYS.

🚫 Revocation & offboarding

Admin Dashboard → Revoke access (plan-gated)

What it is

Actually removing a user from channels — one channel, several, or all of them (offboarding) — with a durable, audited record.

How it works

  • Pick a user and the modal reloads with a multi-select of that user's channels. Choose channels + a reason, or tick Remove from ALL their channels to offboard.
  • The app joins public channels automatically, then removes the user; you get a progress DM and a per-channel result. Idempotent, so a double-submit won't double-kick.
  • Everything is recorded: ACCESS_REVOCATION_INITIATED then ACCESS_REVOKED.
Two Slack constraints, with clear in-app guidance:
• Public channels need the workspace permission Roles & permissions → Account types → "Remove other members from public channels" = Everyone; otherwise Slack returns restricted_action.
• Private channels require the bot to already be a member — run /invite @AccessReview in the channel first.

➕ Access request & approval

Members request Owner approves

What it is

Governed, audited self-service: members ask for channel access and the channel owner approves or denies.

How it works

  • Non-admins open the app and get a member home with ➕ Request channel access (pick a channel + reason).
  • The request is routed to the channel owner (creator, or an active Owner as fallback) as a DM with Approve / Deny.
  • Approve → the app adds the member and logs ACCESS_GRANTED; Deny logs ACCESS_REQUEST_DENIED. The requester is DM'd the outcome.
  • Single-decision (can't be approved twice); only the assigned owner or an admin can decide; already-members are told so.

📊 Governance insights

Admin Dashboard → 📊 Insights

What it is

A one-screen risk read-out that Slack has nothing equivalent to — computed live from the snapshot.

How it works

  • Risk distribution + top-risk channels.
  • Guests & external: guest count, single-channel guests, external users, top external domains.
  • Admin sprawl: admins/owners and where they sit in sensitive channels.
  • Orphaned channels: creator left, or no active admin/owner.
  • Separation of duties: users in both a "request" and an "approval" channel.
  • Policy violations: external users in sensitive channels, private channels with >2 admins, deactivated users still in channels.
  • Remediation queue: remove/flag decisions awaiting enforcement. Review coverage for active campaigns.

📈 Trends & access drift

Admin Dashboard → 📈 Trends

What it is

Change over time — the membership deltas Slack never shows.

How it works

  • A compact snapshot summary is saved periodically (about every 12 hours) to build history.
  • Trends: members, channels, external, guests, and high-risk channels over time, with ▲/▼ change since the previous capture.
  • Drift: since the last snapshot — who joined/left which channels, the change in external users, and newly high-risk channels.
Shows "no history yet" on a fresh install; it fills in over the first day or two of use.

🔒 Tamper-evident audit trail

The evidence backbone.

What it is

Every sensitive action (review decision, revocation, access grant/deny) is written to an append-only, hash-chained log that can't be silently edited.

How it works

  • Each entry stores prev_hash + hash, where the hash is an HMAC-SHA256 over the entry using a secret (AUDIT_HMAC_SECRET) kept outside the database — so an edited or deleted row breaks the chain.
  • Hashing is key-order independent (deterministic canonicalisation), so it survives Postgres JSONB storage.
  • An integrity check runs at boot and periodically. A one-time scripts/reseal-audit-chain.js re-seals existing rows after the canonicalisation fix.

🏷️ Plan detection & gating

Shown in the dashboard header.

What it is

Detects the Slack plan and only offers revoke where it can work.

How it works

  • Enterprise Grid via auth.test; plan tier via team.billing.info (needs the team.billing:read scope).
  • Known Free/Pro → revoke hidden and the per-user modal is read-only. Business+/Enterprise → revoke shown.
  • Fail-open: if the plan can't be read ("Unknown"), revoke stays available (Slack still enforces the real permission). Override anytime with REVOKE_ENABLED=true.

🛡️ Security & data model

How it stays safe and separated.

What it is

A multi-tenant design where each workspace's data is isolated and every privileged action is authorised server-side.

How it works

  • Isolation: per-workspace bot tokens; campaigns, audit log, settings, snapshots, and requests are all keyed by team_id.
  • Authorisation: admin-only actions re-check owner/admin status on every request (fail-closed); reviewers can only act on their assigned channels; approvers only on their requests.
  • Slack signature verification on every request; tokens encrypted at rest in multi-workspace mode.
  • Uninstall removes the token and wipes that workspace's data.
